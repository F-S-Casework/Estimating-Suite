<decisions>
## Implementation Decisions

### Login Screen
- **D-01:** Minimal form — white card, no F&S branding on the login page. Email + password + Sign In button only.
- **D-02:** No registration link, no forgot-password link. Accounts are created and managed exclusively via the Supabase dashboard (Authentication → Users → Invite).
- **D-03:** On failed login, show an inline error message below the form ("Invalid email or password"). No lockout, no redirect.
- **D-04:** After successful login, route to the Home view. Refreshing the browser keeps the user logged in (Supabase session persists via localStorage by default).
- **D-05:** The topbar avatar shows the real signed-in user's initials (replacing the hardcoded "EP"). No other user attribution needed in this phase.

### User Provisioning
- **D-06:** No in-app user management. All 4–10 team members get accounts via Supabase dashboard invite. No self-registration.

### Supabase Configuration
- **D-07:** URL and anon key hardcoded directly in `project/supabase.js`. The anon key is safe to expose — RLS policies protect data at the database level. No config file, no first-run setup screen.

### Bid & Job Numbering
- **D-08:** Bids use the format **Q-YY-NNN** (e.g., Q-26-123). Auto-generated by the database using a per-year sequence.
- **D-09:** When a bid is marked Won, it becomes the next job in the production queue using format **YY-NNN** (e.g., 26-069) — separate sequence from bid numbers. The DB auto-assigns the next available job number.

### Bid Pipeline Stages
- **D-10:** The `bids.stage` column uses exactly these values (in order):
  `ITB` → `Takeoff/Pricing` → `Review` → `Submit` → `Won` → `Lost`

### Job Statuses
- **D-11:** The `jobs.status` column uses exactly these values:
  `Shop`, `Ready`, `Installing`, `Installed`, `Held`

### Contact Roles
- **D-12:** The `contacts.role` column uses: `gc`, `owner`, `sub`, `field`

### Claude's Discretion
- Auth token expiry handling — use Supabase's `onAuthStateChange` listener in the app shell; if the session expires, redirect to login automatically.
- Logout placement — avatar in the topbar opens a small dropdown with a "Sign out" option (already anticipated in the shell design).
- RLS policy approach — single policy per table granting `authenticated` role full read/write access. No per-row ownership policies needed in v1.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `.btn.accent` class — primary action button style; use for the Sign In button
- `.field` + `label` + `input` pattern in `project/styles.css` — already styled form fields; login form uses these directly
- `.card.pad` — centered card on the `--bg` surface; login screen uses this as the form container
- `window.Views` registry — login view registers as `window.Views.login`; App routes to it when unauthenticated
- `window.__go` global — existing navigation; after login, call `window.__go('home')`

### Established Patterns
- CDN-loaded globals: `window.supabase` is exposed by the Supabase JS v2 CDN script; `createClient` is accessed via destructuring `const { createClient } = supabase`
- `useState` / `useEffect` — destructured from `React` at the top of each JSX file (not imported)
- Auth gate in `App` component inside `shell.jsx` — add `session` state, `useEffect` with `supabase.auth.onAuthStateChange`, conditionally render `<LoginView />` or the main `<Rail + content>` layout

### Integration Points
- `project/shell.jsx` → `App()` — auth state lives here; `supabase.auth.getSession()` on mount sets initial state
- `project/index.html` — Supabase CDN tag added before `shell.jsx` script tag; `views-auth.jsx` script tag added after shell
- `supabase/schema.sql` — standalone SQL file run once in Supabase SQL editor; not loaded by the browser

</code_context>

<specifics>
## Specific Ideas

- Bid number format Q-YY-NNN was explicitly stated by the user — e.g., Q-26-123 for a 2026 quote, number 123
- Job numbers are the production queue sequence YY-NNN — completely separate from bid numbers; e.g., a won bid Q-26-123 might become job 26-069 (next in queue)
- The pipeline kanban already shows 5 columns — the DB stages must match the updated 6-stage list (ITB, Takeoff/Pricing, Review, Submit, Won, Lost); the pipeline UI will need to be updated in Phase 3 to reflect this
